Один скрипт, который приводит свежий Ubuntu VPS в рабочее состояние: современные CLI-утилиты, Docker, nginx, zram/swap по проверенной схеме, красивый `fastfetch` + `starship`, и опционально — SSH-hardening с автоматической самопроверкой (никогда не отключает пароль, пока не убедится, что вход по ключу реально работает).

Меню показывает статус всех 15 пунктов сразу — что уже применено, что нет, что настроено не так, как в гайде. Ничего не ставится и не переписывается молча.

## Быстрый старт

```bash
curl -fsSL https://raw.githubusercontent.com/SkyDeaD/UbuntuServer-Fast-Configuration/main/install.sh | sudo bash
```

Скрипт установится как команда **`vsu`** (симлинк в `/usr/local/bin`), сразу проверит обновления и откроет меню. Дальше запускается в любой момент:

```bash
sudo vsu
```

> **Перед первым запуском на боевом сервере** — прогони на тестовой/снэпшотящейся VPS, особенно если планируешь соглашаться на SSH hardening и UFW.

|---|---|---|---|

| 2 | база | CLI-утилиты | `eza`, `bat`, `fd-find`, `ripgrep`, `zoxide`, `ncdu` |
| 3 | база | Базовые пакеты | `micro`, `curl`, `wget`, `git`, `nano`, `certbot`, `python3-certbot-nginx`, `unzip`, `htop`, `bind9-dnsutils`, `jq`, `software-properties-common`, `ca-certificates`, `gnupg`, `rsync` |

| 5 | сервисы | nginx-full | ставит и включает автозапуском |
| 6 | сервисы | fastfetch | PPA `zhangsongcui3371/fastfetch`, версия ≥ 2.64.0 |
| 7 | сервисы | starship | prompt |
| 8 | сервисы | fastfetch config + .bashrc | `fastfetch/config.jsonc` + алиасы/хуки |
| 9 | сервисы | tmux | + минимальный `.tmux.conf` |
| 10 | защита | Docker log rotation | `max-size=10m`, `max-file=3` в `daemon.json` |
| 11 | защита | fail2ban | джейл на реальный SSH-порт |
| 12 | защита | unattended-upgrades | автообновление security-патчей |
| 13 | защита | ZRAM + swap + sysctl + earlyoom | по [гайду](#zram--swap) |
| 14 | защита | SSH hardening | ключи вместо пароля, запрет root-логина, passwordless sudo |
| 15 | защита | UFW | firewall с разрешёнными реально используемыми портами |

`bind9-dnsutils` вместо `dnsutils` — не опечатка: в Ubuntu 26.04 `dnsutils` стал виртуальным пакетом-алиасом, `apt install` его резолвит нормально, а вот `dpkg -s dnsutils` ничего не находит, потому что реально устанавливается `bind9-dnsutils`.

## Как пользоваться меню

```
    #  Пункт                                            Статус
  ──────────────────────────────────────────────────────────────
  ── база ──
    1  Обновление системы                               (действие)
    2  CLI-утилиты (eza/bat/fd/ripgrep/zoxide/ncdu)      ○ не хватает: eza, bat
    3  Базовые пакеты (micro/curl/git/certbot/...)       ✓ установлено
  ── сервисы ──
    4  Docker + Compose                                  ✓ установлен (29.6.1)
    ...
  ── защита и обслуживание ──
   14  SSH hardening                                     ✓ применено (пароль выключен)
   15  UFW firewall                                      ✓ включён
  ──────────────────────────────────────────────────────────────

  5 / 1 3 5 / 1,3,5   применить один пункт или сразу несколько
  A                   применить всё ещё не применённое
  R                   показать команды отката (справочно, ничего не выполняет)
  U                   удалить сам vsu из системы
  Q                   выход
```

- **Один номер** (`5`) — применяет ровно этот пункт, интерактивно, со всеми вопросами внутри как обычно.
- **Несколько номеров** (`1 3 5` или `1,3,5`) — сначала одно общее подтверждение ("применить все выбранные сразу?"), дальше каждый пункт применяется сам, по своим настройкам по умолчанию, без остановки на каждом.
- **Повторный выбор уже применённого пункта** из группы «защита и обслуживание» (Docker log rotation, fail2ban, unattended-upgrades, ZRAM, UFW) — меню само предложит его отключить, отдельного синтаксиса для отмены не нужно.
- **`A`** — то же самое "общее подтверждение → без остановок", но сразу для всего ещё не применённого.
- **`R`** — печатает готовые команды отката для того, что не откатывается автоматически (пакеты, Docker, nginx, fastfetch, starship, конфиги, SSH hardening). Ничего не выполняет сама, только показывает — с явными предупреждениями там, где команда разрушительна (Docker удаляет volume'ы, nginx — конфиги сайтов, снятие SSH hardening — возврат к паролю).
- **`U`** — удаляет сам `vsu` (`/opt/vps-setup` + команду) из системы. Ничего из установленного на систему не трогает.

## Как это работает

### Идемпотентность

Каждый пункт меню — пара функций: `status_*` (только читает состояние, ничего не меняет) и `apply_*`/`disable_*` (действие). Меню перед каждым показом заново опрашивает `status_*` всех пунктов:

- пакет уже установлен → `✓`, повторный выбор ничего лишнего не переспросит;
- блок в `.bashrc` уже есть (по маркеру `# >>> vps-setup >>>` / `# <<< vps-setup <<<`) → не дублируется;
- `zram`/`swapfile`/`sysctl` настроены **как в гайде** → `✓`;
- настроены **иначе** (кто-то — скорее всего ты сам — уже покрутил руками) → `!`, и при выборе пункта скрипт покажет разницу и спросит явно, по умолчанию **не** перезаписывая.

Своп определяется по принципу "любой не-zram своп", а не по конкретному имени файла — `/swapfile`, `/swap.img` и другие варианты распознаются одинаково корректно.

### SSH hardening с самопроверкой

Самый рискованный пункт — отключение пароля и root-логина по SSH. Ошибка здесь означает потерю доступа к серверу, если нет консоли у провайдера. Поэтому пункт 14 работает не "накатил и надейся":

1. Просит вставить публичный ключ (или использует уже имеющийся в `authorized_keys`).
2. Генерирует **одноразовый тестовый ключ прямо на сервере**, временно добавляет его и реально логинится через `ssh user@127.0.0.1` — **до** каких-либо изменений `sshd_config`.
3. Только если базовый вход подтверждён — пишет `/etc/ssh/sshd_config.d/10-hardening.conf` (лексически раньше `50-cloud-init.conf`, который cloud-init может пересоздавать), проверяет `sshd -t`, перезапускает `sshd`.
4. **Проверяет вход по ключу ещё раз, уже после рестарта.** Если не проходит — автоматический откат.
5. Тестовый ключ в конце всегда удаляется — остаётся только твой реальный.

Текущая SSH-сессия не разрывается ни на одном этапе: рестарт `sshd` не убивает уже открытые соединения, обрывает только приём новых.

### UFW без сюрпризов

Если на сервере уже крутится VPN/прокси на нестандартном порту, включение firewall с разрешённым только 22-м портом молча его оборвёт. Пункт 15 сначала показывает список реально слушающих TCP-портов, и только потом спрашивает — по умолчанию **нет**.

### ZRAM + swap

```
RAM
 │
 ▼
zram (~75% RAM, lz4, priority 100)      ← используется первым
 │
 ▼
swap на диске (1 GB, priority 10)       ← только если zram уже забит
 │
 ▼
earlyoom (опционально)                  ← подчищает самый прожорливый процесс
                                            до того, как это вслепую сделает OOM killer
```

Плюс `sysctl`: `vm.swappiness=80` и `vm.vfs_cache_pressure=50`.

## Кастомизация

Свой конфиг `fastfetch` — отредактируй `config.jsonc` и закоммить, `setup.sh` подтягивает его напрямую по сырой ссылке при каждом запуске.

Свой форк — поменяй `REPO_RAW_BASE` в начале `install.sh` и `setup.sh`:

```bash
REPO_RAW_BASE="https://raw.githubusercontent.com/<твой-юзер>/<твой-форк>/<ветка-или-тег>"
```

**Важно:** при любой правке `setup.sh` поднимай номер в файле `VERSION` — именно по нему `vsu` понимает, что вышло обновление. Если забыть — уже установленные копии скрипта не увидят изменений.

## Требования

- Ubuntu 24.04 или 26.04 (для других версий не тестировалось)
- root-доступ (`sudo`)
- исходящий доступ в интернет (apt, PPA, GitHub raw, download.docker.com, starship.rs)

## Сознательно не автоматизировано

- Смена SSH-порта с 22 на нестандартный
- Блокировка пароля root (`passwd -l root`) — при отключённом `PermitRootLogin` избыточно
- Более тонкие правила UFW (rate-limiting, конкретные IP)
- Настройка самого VPN/прокси-стека — это отдельная задача под конкретный сервер

## FAQ / известные грабли

**После установки конфигов новые алиасы не появились.**
Скрипт работает в своём подпроцессе — `source ~/.bashrc`, выполненный ИЗ скрипта, не может повлиять на твою текущую интерактивную сессию (дочерний процесс не меняет окружение родительского). Выполни `source ~/.bashrc` сам, в своём терминале, либо просто переподключись по SSH.

**Docker не ставится, ошибка про кодовое имя дистрибутива.**
Официальный репозиторий Docker иногда отстаёт от свежих релизов Ubuntu. Скрипт сам детектит отсутствие пакетов под текущий коднейм и переключается на `noble` — пакеты совместимы.

**После zram-пункта `swapon --show` показывает не то, что ожидалось.**
Проверь `journalctl -u zramswap` — часто причина в том, что старое устройство не освободилось. Помогает `sudo swapoff -a && sudo systemctl restart zramswap`.

**Хочу пересмотреть решение по SSH hardening/UFW, не пересоздавая сервер.**
`sudo vsu` ещё раз — на уже настроенных пунктах скрипт покажет текущее состояние и спросит, менять ли.

**`/etc/sysctl.conf` отсутствует на сервере.**
Нормально для многих облачных образов. Скрипт использует `/etc/sysctl.d/99-zram.conf` — это официально рекомендуемый способ в современных Ubuntu/Debian.
