## Быстрый старт

```bash
curl -fsSL https://raw.githubusercontent.com/SkyDeaD/UbuntuServer-Fast-Configuration/main/install.sh | sudo bash
```

Ставится как команда `usfc` (симлинк в `/usr/local/bin`), сразу открывает меню. Дальше — просто `usfc`, `sudo` уже не нужен: пункт 6 сам добавляет `alias usfc='sudo usfc'` в `.bashrc`.

В меню — либо номер пункта (`5`, можно несколько через пробел или запятую: `1 3 5`), либо весь раздел разом: `B` — база, `S` — сервисы, `P` — защита и обслуживание, `A` — вообще всё, можно комбинировать (`B,S`, `5,P`). На всё, что применяешь пачкой, спрашивается один раз, дальше каждый пункт идёт по своим дефолтным ответам без остановок.

> Перед боевым сервером советую прогнать на тестовой VPS хотя бы раз — не потому что что-то обязательно сломается, а потому что пункт SSH hardening в теории может тебя же и запереть, если что-то в сети пойдёт не так во время самопроверки.

<summary>Подробнее по каждому разделу</summary>

**База** — `apt update/upgrade`; базовый набор пакетов (`micro`, `curl`, `wget`, `git`, `nano`, `certbot` + `python3-certbot-nginx`, `unzip`, `htop`, `jq`, `rsync` и ещё несколько вещей, которые обычно ставишь в первую же минуту на любом сервере — в том числе `software-properties-common`, без которого не заработает `add-apt-repository`, нужный дальше для PPA fastfetch); современные замены классических утилит (`eza` вместо `ls` с иконками, `bat`/`batcat` вместо `cat` с подсветкой, `fd`/`fdfind` вместо `find`, `ripgrep`, `zoxide` — «умный» `cd`, который запоминает частые директории, `ncdu` для разбора места на диске).

**Сервисы** — fastfetch версии не ниже 2.64.0 (важно — более старые не умеют выравнивание в format-строках, которое использует прилагаемый `config.jsonc`); starship; сам конфиг fastfetch + алиасы в `.bashrc` — этот пункт идёт сразу за fastfetch/starship/CLI-утилитами не просто так: он ссылается на них в `eval`-строках и алиасах, и логичнее применять конфиг сразу, пока утилиты уже стоят; tmux с минимальным конфигом (мышь, история на 10000 строк); Docker CE + Compose plugin из официального репозитория Docker (не пакет `docker.io` из репов Ubuntu — он старый); nginx-full.

**Защита и обслуживание** — ротация логов Docker (лимит 10 МБ на файл, иначе `journalctl`/`docker logs` на маленьком VPS сожрут диск за пару недель шумного контейнера); fail2ban на реальный SSH-порт сервера (не хардкожен 22); unattended-upgrades; связка zram (lz4, 75% RAM, приоритет 100) + swapfile на диске (1 GB, приоритет 10) + `vm.swappiness=80` и `vm.vfs_cache_pressure=50` через `/etc/sysctl.d/99-zram.conf` + опционально earlyoom; SSH hardening; UFW. Раздел идёт последним осознанно — UFW при включении сканирует реально слушающие порты, и если Docker/nginx из раздела «сервисы» уже подняты, фаервол сразу увидит их порты, а не только 22-й.

Подробности по каждому пункту — прямо в меню. Клавиша `H` там же — справка по алиасам, которые скрипт ставит (`ls`/`ll`/`la`/`lt`/`cat`/`catp`/`scat`/`fd`).

## Как это работает

Каждый пункт меню — это пара функций: одна только проверяет текущее состояние сервера, вторая применяет. Перед каждой перерисовкой меню все статусы пересчитываются заново, поэтому если что-то уже настроено — в том числе не этим скриптом, а руками до него — это видно сразу, и повторный выбор пункта не будет бездумно перезаписывать конфиг.

<details>
<summary>Как именно работает самопроверка SSH hardening</summary>

Самый рискованный пункт, поэтому единственный, где скрипт не просто применяет конфиг и надеется. Перед тем как выключить пароль, он генерирует одноразовый ключ прямо на сервере, добавляет его во временные `authorized_keys` и реально логинится через `ssh user@127.0.0.1` — сначала до изменения `sshd_config`, чтобы убедиться, что база вообще работает, потом ещё раз после рестарта `sshd` с новым конфигом. Не прошло — автоматический откат: удаляется `/etc/ssh/sshd_config.d/10-hardening.conf`, `sshd` перезапускается со старыми настройками, пароль остаётся включён.

Файл назван `10-hardening.conf` не просто так — конфиги в `sshd_config.d/` читаются по алфавиту, и на многих облачных образах уже лежит `50-cloud-init.conf` с `PasswordAuthentication yes`, который cloud-init может пересоздать при пересборке инстанса. `10` идёт раньше `50`, так что наши настройки побеждают при слиянии.

Текущая SSH-сессия при этом не рвётся — рестарт `sshd` не убивает уже открытые соединения, только перестаёт принимать новые, и это как раз то, что позволяет тестировать вслепую, не теряя доступ.

</details>

<details>
<summary>Почему UFW не ломает уже работающие сервисы</summary>

Перед включением скрипт парсит `ss -tln` и показывает список реально слушающих портов — если на сервере уже висит что-то на нестандартном порту (VPN, прокси), это будет видно до того, как фаервол его отрежет, а не после.

</details>

## Кастомизация

`config.jsonc` для fastfetch — редактируешь и коммитишь, `setup.sh` тянет его по сырой ссылке при каждом запуске, локально ничего хранить не нужно.

Свой форк — меняешь `REPO_RAW_BASE` в начале `install.sh` и `setup.sh` на свой репозиторий. Версия проверяется через отдельный файл `VERSION` — если правишь `setup.sh`, не забудь его поднять, иначе уже установленные копии не увидят, что вышло обновление.

## Сознательно не автоматизировано

Смена SSH-порта — слишком специфично под конкретный сетап. Блокировка пароля root (`passwd -l root`) — избыточно, если `PermitRootLogin` уже `no`. Тонкая настройка UFW (rate-limiting, конкретные IP) и сама настройка VPN/прокси-стека — отдельные задачи под конкретный сервер, не generic bootstrap.

## FAQ

<details>
<summary>Новые алиасы не появились сразу после установки конфигов</summary>

Скрипт работает в дочернем процессе — `source ~/.bashrc`, выполненный из него, физически не может повлиять на твою текущую SSH-сессию, дочерний процесс не меняет окружение родительского, это ограничение Unix, не обходится никаким трюком внутри скрипта. В конце меню сам об этом напоминает — выполни `source ~/.bashrc` сам, либо просто переподключись по SSH.

</details>

<details>
<summary>Docker не ставится, ошибка про кодовое имя дистрибутива</summary>

Официальный репозиторий Docker иногда отстаёт от свежих релизов Ubuntu на пару недель. Скрипт сам детектит отсутствие пакетов под текущий коднейм (`apt-cache policy docker-ce-cli`) и переключается на `noble` — пакеты полностью совместимы.

</details>

<details>
<summary>Хочу пересмотреть SSH hardening или UFW без пересоздания сервера</summary>

`usfc` ещё раз — на уже настроенных пунктах меню покажет текущее состояние и спросит, менять ли, а не сделает это молча.

</details>
