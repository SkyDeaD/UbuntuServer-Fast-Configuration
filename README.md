Один скрипт, который приводит свежий Ubuntu VPS в рабочее состояние: современные CLI-утилиты, Docker, zram/swap по проверенной схеме, красивый `fastfetch` + `starship`, и опционально — SSH-hardening с автоматической самопроверкой (никогда не отключает пароль, пока не убедится, что вход по ключу реально работает).

Каждый пункт — отдельный вопрос **да/нет**. Ничего не ставится и не переписывается молча. Если что-то уже настроено (в том числе вручную, не этим скриптом) — скрипт это увидит и не тронет без явного подтверждения.

## Быстрый старт

```bash
curl -fsSL https://raw.githubusercontent.com/SkyDeaD/UbuntuServer-Fast-Configuration/main/install.sh | sudo bash
```

Скрипт установится как команда `vps-setup` (симлинк в `/usr/local/bin`), сразу проверит обновления и откроет меню. Дальше запускается в любой момент:

```bash
vps-setup
```

При каждом запуске `vps-setup` сначала сверяет свою версию с [`VERSION`](VERSION) в репозитории — если есть новее, предложит обновиться и перезапустится уже новой версией. Меню показывает статус всех 15 пунктов сразу (✓ применено / ○ не применено / ! отличается от рекомендованного), выбор пункта по номеру применяет его, `A` — применяет всё ещё не применённое разом, `d<номер>` — отключает то, что безопасно отключать (см. [ниже](#идемпотентность)).

> **Перед первым запуском на боевом сервере** — прогони на тестовой/снэпшотящейся VPS, особенно если планируешь соглашаться на SSH hardening и UFW. Скрипт написан максимально осторожно (см. [ниже](#ssh-hardening-с-самопроверкой)), но это по-прежнему root-доступ и сетевые правила.

## Как это работает

### Идемпотентность

Каждый из 14 пунктов меню — не разовое действие, а пара функций: `status_*` (только читает текущее состояние системы, ничего не меняет) и `apply_*` (действие). Меню перед каждым показом заново опрашивает `status_*` всех пунктов, поэтому:

- пакет уже установлен → в меню сразу видно `✓`, повторный выбор пункта просто не будет тебя переспрашивать заново то, что уже стоит;
- блок в `.bashrc` уже есть (по маркеру `# >>> vps-setup >>>` / `# <<< vps-setup <<<`) → не дублируется;
- `zram`/`swapfile`/`sysctl` настроены **как в гайде** → `✓`;
- настроены **иначе, чем в гайде** (то есть кто-то — скорее всего ты сам — уже покрутил руками) → `!`, и при выборе пункта скрипт явно покажет разницу и спросит подтверждение, по умолчанию **не** перезаписывая.

**Отмена (`d<номер>`)** поддерживается только для того, что безопасно выключать одним действием: Docker log rotation, fail2ban, unattended-upgrades, zram, UFW. Для остального (установка пакетов, Docker, SSH hardening) автоматический откат через меню не сделан намеренно — это либо неопасно оставить как есть, либо слишком рискованно откатывать одной кнопкой; меню в этом случае подскажет, что делать руками.

### SSH hardening с самопроверкой

Самый рискованный шаг во всём скрипте — отключение пароля и root-логина по SSH. Ошибка здесь означает потерю доступа к серверу, если нет консоли у провайдера. Поэтому шаг 13 работает не "накатил и надейся", а с фактической проверкой на живую:

1. Просит вставить публичный ключ (или использует уже имеющийся в `authorized_keys`).
2. Генерирует **одноразовый тестовый ключ прямо на сервере**, временно добавляет его и реально логинится через `ssh user@127.0.0.1` — **до** каких-либо изменений `sshd_config`. Это ловит проблемы с правами на `~/.ssh` до того, как что-то стало необратимым.
3. Только если базовый вход подтверждён — пишет отдельный `/etc/ssh/sshd_config.d/10-hardening.conf` (имя специально выбрано так, чтобы лексически идти раньше `50-cloud-init.conf`, который cloud-init может пересоздавать при пересборке образа и включает в себя `PasswordAuthentication yes` — реальная находка при тестировании на живых VPS), проверяет синтаксис через `sshd -t`, перезапускает `sshd`.
4. **Проверяет вход по ключу ещё раз, уже после рестарта.** Если не проходит — автоматический откат: `10-hardening.conf` удаляется, `sshd` перезапускается со старым конфигом, пароль остаётся включён.
5. Одноразовый тестовый ключ в конце всегда удаляется из `authorized_keys` — остаётся только твой реальный.

На всех этапах текущая SSH-сессия не разрывается: рестарт `sshd` не убивает уже открытые соединения, обрывает только приём новых — это и позволяет тестировать в фоне, не теряя доступ, даже если что-то пойдёт не так.

Если скрипт запущен напрямую под root (без обычного пользователя через `sudo`) — шаг честно откажется от hardening: закрывать root-логин, когда кроме root пускать больше некого, означает гарантированно себя же и запереть.

### UFW без сюрпризов

Если на сервере уже крутится VPN/прокси на нестандартном порту (частый случай), включение firewall с разрешённым только 22-м портом молча оборвёт эти сервисы. Поэтому шаг 14 сначала выводит список реально слушающих TCP-портов (`ss -tln`), и только потом спрашивает — по умолчанию **нет**. При согласии разрешает SSH-порт и все обнаруженные порты, а не только 22.

### ZRAM + swap

Схема двухуровневая: `zram` (сжатый swap в RAM, быстрый) как основной уровень + `swapfile` на диске как резервная подушка на случай, если `zram` переполнится.

```
RAM
 │
 ▼
zram (~75% RAM, lz4, priority 100)      ← используется первым
 │
 ▼
swapfile на диске (1 GB, priority 10)   ← только если zram уже забит
 │
 ▼
earlyoom (опционально)                  ← подчищает самый прожорливый процесс
                                            до того, как это вслепую сделает OOM killer
```

Плюс `sysctl`: `vm.swappiness=80` (система активнее уходит в zram — это дёшево, это RAM) и `vm.vfs_cache_pressure=50` (баланс с файловым кэшем на маленьком VPS).

## Кастомизация

Свой конфиг `fastfetch` — просто отредактируй `config.jsonc` в репозитории и закоммить, `setup.sh` подтягивает его напрямую по сырой ссылке при каждом запуске.

Свой форк — поменяй `REPO_RAW_BASE` в начале `install.sh` и `setup.sh`:

```bash
REPO_RAW_BASE="https://raw.githubusercontent.com/<твой-юзер>/<твой-форк>/<ветка-или-тег>"
```

Рекомендуется указывать конкретный тег, а не `main` — так конфиг можно обновлять в репозитории, не боясь неожиданно поменять то, что уже накатано на прод-серверах.

**Важно:** при любой правке `setup.sh` в репозитории не забывай поднимать номер в файле `VERSION` — именно по нему `vps-setup` понимает, что вышло обновление. Если `VERSION` не поднять, уже установленные копии скрипта не увидят изменений.

## Требования

- Ubuntu 24.04 или 26.04 (для других версий не тестировалось)
- root-доступ (`sudo`)
- исходящий доступ в интернет (apt, PPA, GitHub raw, download.docker.com, starship.rs)

## Сознательно не автоматизировано

- Смена SSH-порта с 22 на нестандартный
- Блокировка пароля root (`passwd -l root`) — при отключённом `PermitRootLogin` избыточно
- Более тонкие правила UFW (rate-limiting, конкретные IP)
- Настройка самого VPN/прокси-стека (Xray, MTProto и т.п.) — это отдельная задача под конкретный сервер, не общий бутстрап

## FAQ / известные грабли

**Docker не ставится, ошибка про кодовое имя дистрибутива.**
Официальный репозиторий Docker иногда отстаёт от свежих релизов Ubuntu на несколько недель. Скрипт сам детектит отсутствие пакетов под текущий коднейм и переключается на `noble` (24.04) — пакеты совместимы. Если и это не помогло, проверь вручную: `apt-cache policy docker-ce-cli`.

**После `zram`-шага `swapon --show` показывает не то, что ожидалось.**
Проверь `journalctl -u zramswap` — часто причина в том, что старое устройство не освободилось. Помогает `sudo swapoff -a && sudo systemctl restart zramswap`.

**Хочу пересмотреть решение по SSH hardening/UFW, не пересоздавая сервер.**
Просто `sudo vps-setup` ещё раз — на уже настроенных пунктах скрипт покажет текущее состояние и спросит, менять ли.

**`/etc/sysctl.conf` отсутствует на сервере.**
Нормально для многих облачных образов (Oracle Cloud и др.). Скрипт использует `/etc/sysctl.d/99-zram.conf` — это официально рекомендуемый способ в современных Ubuntu/Debian, ничего создавать в `/etc/sysctl.conf` не нужно.
