Каждый раз, когда поднимаешь новый VPS, повторяется одно и то же: поставить нормальный `ls`, накатить Docker, разобраться со свопом на маленьком сервере, включить SSH только по ключу и не забыть про UFW. Чтобы не повторять регулярно одно и то же — написал скрипт с меню на 12 пунктов, который делает это всё сам, спрашивая только там, где решение реально важно принять тебе, а не ему.

## Быстрый старт

```bash
curl -fsSL https://raw.githubusercontent.com/SkyDeaD/UbuntuServer-Fast-Configuration/main/install.sh | sudo bash && source ~/.bashrc
```

Ставится как команда `usfc`, сразу тихо делает `apt update` и открывает меню. Функция `usfc()` (не alias) добавляется в `.bashrc` сама при первом запуске — дальше просто `usfc`, `sudo` руками писать не нужно, а когда меню закрывается, она сама подтягивает `.bashrc` в твою же сессию, так что новые алиасы/промпт видны сразу, без ручного `source` или переподключения (подробнее — в FAQ).

Хвостовой `&& source ~/.bashrc` в команде выше — не для красоты: сам скрипт запускается через `sudo` и не может повлиять на твою оболочку изнутри (см. FAQ), а вот `&&` выполняется уже в твоём собственном шелле — поэтому именно на самой первой установке (когда функции `usfc` в `.bashrc` ещё не существует) алиасы всё равно подхватятся сразу, без переподключения.

```
  ┌─────┬────────────┬────────────────────────────┬────────────────────────────┐
  │ #   │ Раздел     │ Пункт                      │ Статус                     │
  ├─────┼────────────┼────────────────────────────┼────────────────────────────┤
  │ 1   │ база       │ Базовые пакеты             │ ○ не хватает: micro, curl… │
  │ 2   │ база       │ CLI-утилиты + starship     │ ✓ установлено              │
  │ 3   │ база       │ fastfetch                  │ ✓ 2.66.0                   │
  │ 4   │ база       │ tmux                       │ ! установлен, конфига нет  │
  │ 5   │ сервисы    │ Docker + Compose           │ ○ не установлен            │
  │ 6   │ сервисы    │ nginx-full                 │ ○ не установлен            │
  │ 7   │ защита     │ Docker log rotation        │ — (нужен Docker)           │
  │ 8   │ защита     │ fail2ban                   │ ○ не запущен               │
  │ 9   │ защита     │ unattended-upgrades        │ ✓ включено                 │
  │ 10  │ защита     │ ZRAM + swap + earlyoom     │ ○ не настроено             │
  │ 11  │ защита     │ SSH hardening              │ ○ не применено             │
  │ 12  │ защита     │ UFW firewall               │ ○ выключен                 │
  └─────┴────────────┴────────────────────────────┴────────────────────────────┘

  ┌────────────────────────────────────────────────────────────────────────────┐
  │ Выбор:    5 / 1 3 5 / 1,3,5 — один или несколько пунктов сразу             │
  │           буквы разделов тоже можно сочетать (B,S); применённый пункт «за… │
  │ Разделы:  B база          S сервисы       P защита        A всё            │
  │ Команды:  H алиасы        R откат         U удалить       Q выход          │
  └────────────────────────────────────────────────────────────────────────────┘
```

В меню — номер пункта (`5`, можно несколько сразу: `1 3 5` или `1,3,5`), раздел целиком (`B`/`S`/`P`), всё сразу (`A`), можно сочетать (`B,S`). На пачку из нескольких пунктов вопрос один, дальше каждый идёт по своим дефолтным ответам без остановок. `H` — справка по алиасам, `R` — команды отката, `U` — снести сам `usfc`.

Не доверяешь `curl | sudo bash` и хочешь применить то же самое руками, пункт за пунктом? — держи [ручную инструкцию](docs/MANUAL.md).

<details>
<summary>Подробнее по каждому пункту</summary>

**Базовые пакеты** — `micro`, `curl`, `wget`, `git`, `nano`, `certbot` + `python3-certbot-nginx`, `unzip`, `htop`, `jq`, `rsync` и ещё несколько вещей, которые обычно ставишь в первую же минуту на любом сервере — в том числе `software-properties-common`, без которого не заработает `add-apt-repository`, нужный дальше для PPA fastfetch. Специально идёт первым пунктом.

**CLI-утилиты + starship** — современные замены классических утилит (`eza` вместо `ls` с иконками, `bat`/`batcat` вместо `cat` с подсветкой, `fd`/`fdfind` вместо `find`, `ripgrep`, `zoxide` — «умный» `cd`, `ncdu` для разбора места на диске) и starship-промпт — всё вместе, потому что это один и тот же слой "как выглядит и ощущается терминал". Алиасы в `.bashrc` (`ls`/`ll`/`la`/`lt`/`cat`/`catp`/`scat`/`fd`) и `eval`-строки для zoxide/starship пишутся сразу тем же пунктом, а не отдельным шагом позже.

**fastfetch** — версия не ниже 2.64.0 (более старые не умеют выравнивание в format-строках, которое использует прилагаемый `config.jsonc`). Конфиг и автозапуск при входе по SSH пишутся в `.bashrc` сразу тем же пунктом — не нужно ставить fastfetch, а потом отдельно вспоминать про конфиг.

**tmux** — с минимальным конфигом (мышь, история на 10000 строк).

**Docker + Compose** — CE + Compose plugin из официального репозитория Docker (не пакет `docker.io` из репов Ubuntu — он старый).

**nginx-full**, **Docker log rotation** (лимит 10 МБ на файл), **fail2ban** (на реальный SSH-порт сервера, не хардкожен 22), **unattended-upgrades**, **ZRAM + swap** (zram lz4 75% RAM приоритет 100 + swapfile на диске 1 GB приоритет 10 + `vm.swappiness=80`/`vm.vfs_cache_pressure=50` + опционально earlyoom), **SSH hardening**, **UFW** — раздел «защита» идёт последним осознанно: UFW при включении сканирует реально слушающие порты, и если Docker/nginx уже подняты, фаервол сразу увидит их порты, а не только 22-й.

Обновление системы (`apt upgrade`) не вынесено отдельным пунктом — это решение оставлено тебе, `apt update` (просто обновить списки пакетов) скрипт делает сам один раз при запуске.

</details>

## Как это работает

Каждый пункт меню — это пара функций: одна только проверяет текущее состояние сервера, вторая применяет. Перед каждой перерисовкой меню все статусы пересчитываются заново, поэтому если что-то уже настроено — в том числе не этим скриптом, а руками до него — это видно сразу, и повторный выбор пункта не будет бездумно перезаписывать конфиг.

<details>
<summary>Как именно работает самопроверка SSH hardening</summary>

Самый рискованный пункт, поэтому единственный, где скрипт не просто применяет конфиг и надеется. Перед тем как выключить пароль, он генерирует одноразовый ключ прямо на сервере, добавляет его во временные `authorized_keys` и реально логинится через `ssh user@127.0.0.1` — сначала до изменения `sshd_config`, чтобы убедиться, что база вообще работает, потом ещё раз после рестарта `sshd` с новым конфигом. Не прошло — автоматический откат: удаляется `/etc/ssh/sshd_config.d/10-hardening.conf`, `sshd` перезапускается со старыми настройками, пароль остаётся включён.

Файл назван `10-hardening.conf` не просто так — конфиги в `sshd_config.d/` читаются по алфавиту, и на многих облачных образах уже лежит `50-cloud-init.conf` с `PasswordAuthentication yes`, который cloud-init может пересоздать при пересборке инстанса. `10` идёт раньше `50`, так что наши настройки побеждают при слиянии.

Текущая SSH-сессия при этом не рвётся — рестарт `sshd` не убивает уже открытые соединения, только перестаёт принимать новые.

</details>

<details>
<summary>Почему UFW не ломает уже работающие сервисы</summary>

Перед включением скрипт парсит `ss -tln` и показывает список реально слушающих портов — если на сервере уже висит что-то на нестандартном порту (VPN, прокси), это будет видно до того, как фаервол его отрежет, а не после.

</details>

## Кастомизация

`src/config.jsonc` для fastfetch — редактируешь и коммитишь, `src/setup.sh` тянет его по сырой ссылке при каждом запуске.

Свой форк — меняешь `REPO_RAW_BASE` в начале `install.sh` и `src/setup.sh`. Версия проверяется через отдельный файл `src/VERSION` — если правишь `src/setup.sh`, не забудь его поднять.

## Сознательно не автоматизировано

`apt upgrade` — решаешь сам, когда. Смена SSH-порта — слишком специфично под конкретный сетап. Блокировка пароля root (`passwd -l root`) — избыточно, если `PermitRootLogin` уже `no`. Тонкая настройка UFW (rate-limiting, конкретные IP) и сама настройка VPN/прокси-стека — отдельные задачи под конкретный сервер.

## FAQ

<details>
<summary>Новые алиасы не появились сразу после установки</summary>

Если ставил командой из «Быстрого старта» как есть (с `&& source ~/.bashrc` на конце) — не должно было случиться: этот хвост выполняется в твоей собственной оболочке и подхватывает `.bashrc` сразу же, даже на самой первой установке. Начиная со второго запуска `usfc` (уже после того как в `.bashrc` появилась функция-обёртка) она и без этого хвоста сама делает `source ~/.bashrc` сразу после выхода из меню.

Если всё же не появились — скорее всего, ставил без `&& source ~/.bashrc` (например, скопировал только часть команды). Скрипт запускается через `sudo` и не может повлиять на твою текущую оболочку изнутри — это ограничение Unix, не баг. Выполни `source ~/.bashrc` сам один раз, либо переподключись по SSH — дальше `usfc` подхватывает всё сам.

</details>

<details>
<summary>Docker не ставится, ошибка про кодовое имя дистрибутива</summary>

Официальный репозиторий Docker иногда отстаёт от свежих релизов Ubuntu — скрипт сам это ловит и переключается на совместимый `noble`.

</details>

<details>
<summary>Хочу пересмотреть SSH hardening или UFW без пересоздания сервера</summary>

`usfc` ещё раз — на уже настроенных пунктах меню покажет текущее состояние и спросит, менять ли.

</details>
